📦 Volume Mount Chain
Host: ./workspaces/
   ↳ project-service mounts → /workspaces (writes scaffold files)
   ↳ sandbox-worker mounts → /workspaces (reads project files)
   ↳ repl-<id> uses --volumes-from sandbox-worker → gets /workspaces/<id> as /workspace
   ↳ execution-service uses container.exec() → opens /bin/sh inside repl-<id>
🤔 Why --volumes-from sandbox-worker?
The Problem: sandbox-worker spawns containers by calling the host Docker daemon via /var/run/docker.sock. When it runs:

   docker run -v /workspaces/<id>:/workspace mini-replit-node

The path /workspaces/<id> is a path inside the sandbox-worker container — but Docker resolves volume paths on the host machine, not inside the container. So the host has no idea what /workspaces/<id> means and the mount silently fails or errors.

On Windows it gets worse: the real host path is something like
   D:/coding/development/Web Development/100x/.../workspaces/<id>
The spaces in the path break shell commands, and Linux containers don't understand Windows drive letters at all.

The Fix — --volumes-from sandbox-worker:
Instead of specifying a path, we tell Docker: "give this new container the exact same volume mounts that sandbox-worker already has." Since Docker Compose already resolved ./workspaces → /workspaces correctly when it started sandbox-worker, the new container inherits that binding without us ever needing to know or pass the host path. This works on Windows, Mac, and Linux with zero changes.

Why container_name: sandbox-worker is required:
--volumes-from needs a stable container name to reference. Without it, Docker Compose generates a random name like replit-sandbox-worker-1 which could change on restart. Setting container_name: sandbox-worker guarantees the name is always predictable.
